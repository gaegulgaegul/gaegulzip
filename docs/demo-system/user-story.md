# 사용자 스토리: SDK 데모 시스템

## 개요

개발자가 SDK 패키지(qna, notice)의 UI/UX를 실제 앱 환경에서 검증하고, Design System과의 통합 동작을 확인할 수 있는 데모 시스템을 제공합니다. 기존 design_system_demo 앱을 확장하여 SDK 기능을 시각적으로 테스트하고, 다양한 시나리오를 빠르게 실험할 수 있는 환경을 구축함으로써 개발 및 QA 효율을 높입니다.

## 사용자 스토리

### US-1: SDK 데모 카테고리 추가

- **As a** 개발자
- **I want to** 홈 화면에서 "SDK Demos" 카테고리를 선택할 수 있다
- **So that** Design System 컴포넌트와 SDK 기능을 동일한 앱에서 테스트할 수 있다

### US-2: QnA SDK 데모 페이지 접근

- **As a** 개발자
- **I want to** SDK Demos 카테고리에서 "QnA" 항목을 선택하여 질문 제출 기능을 테스트할 수 있다
- **So that** QnA 패키지의 UI, 유효성 검증, 에러 처리가 올바르게 동작하는지 확인할 수 있다

### US-3: Notice SDK 데모 페이지 접근

- **As a** 개발자
- **I want to** SDK Demos 카테고리에서 "Notice" 항목을 선택하여 공지사항 목록/상세 기능을 테스트할 수 있다
- **So that** Notice 패키지의 목록 표시, 페이지네이션, 읽음 상태 추적이 올바르게 동작하는지 확인할 수 있다

### US-4: 독립적인 SDK 테스트 환경

- **As a** 개발자
- **I want to** 각 SDK 데모 페이지가 독립적으로 동작한다
- **So that** 서버 연동 없이도 UI/UX를 검증하고, 다양한 상태(로딩, 에러, 빈 데이터)를 시뮬레이션할 수 있다

### US-5: Design System과의 통합 확인

- **As a** 개발자
- **I want to** SDK 위젯들이 Design System의 테마(라이트/다크)와 스타일을 올바르게 적용한다
- **So that** SDK가 어떤 앱에 통합되어도 일관된 디자인을 유지하는지 확인할 수 있다

## 사용자 시나리오

### 시나리오 1: QnA SDK 데모 접근 및 질문 제출 테스트

1. 개발자가 데모 앱을 실행한다
2. 홈 화면에 "SDK Demos" 카테고리 카드가 표시된다
3. 개발자가 "SDK Demos" 카드를 탭한다
4. SDK 목록 화면에 "QnA", "Notice" 항목이 표시된다
5. 개발자가 "QnA" 항목을 탭한다
6. QnA 질문 제출 화면(QnaSubmitView)이 표시된다
7. 개발자가 제목과 본문을 입력한다
8. 개발자가 "질문 제출" 버튼을 탭한다
9. 결과: 제출 처리가 실행되고, 성공/실패 모달이 표시된다 (실제 서버 연동 여부와 무관하게 UI 동작 확인 가능)

### 시나리오 2: Notice SDK 데모 접근 및 공지사항 목록 확인

1. 개발자가 SDK 목록 화면에서 "Notice" 항목을 탭한다
2. 공지사항 목록 화면(NoticeListView)이 표시된다
3. 화면에 고정 공지와 일반 공지가 구분되어 표시된다
4. 개발자가 스크롤하여 페이지네이션 동작을 확인한다
5. 개발자가 공지 항목을 탭한다
6. 공지사항 상세 화면(NoticeDetailView)이 표시된다
7. 결과: 공지 내용이 마크다운 형식으로 렌더링되고, 읽음 상태가 업데이트된다

### 시나리오 3: 테마 변경 시 SDK 위젯 스타일 확인

1. 개발자가 QnA 또는 Notice 데모 화면에 있다
2. 개발자가 앱바의 테마 전환 버튼을 탭한다
3. 화면이 라이트/다크 모드로 전환된다
4. SDK 위젯들(SketchInput, SketchButton, NoticeListCard 등)이 테마에 맞게 색상과 스타일이 변경된다
5. 결과: SDK 위젯들이 Design System의 테마 시스템을 올바르게 따른다

### 시나리오 4: 다양한 상태 시뮬레이션

1. 개발자가 Notice 데모 화면을 연다
2. 초기 로딩 상태가 표시된다 (로딩 인디케이터 + 안내 메시지)
3. 데이터가 로드되면 목록이 표시된다
4. 네트워크 에러 시뮬레이션 시 에러 상태 화면이 표시된다 (아이콘 + 메시지 + 재시도 버튼)
5. 빈 데이터 시뮬레이션 시 빈 상태 화면이 표시된다 (아이콘 + 안내 메시지)
6. 결과: 각 상태별 UI가 올바르게 표시되고, 사용자 피드백이 명확하다

## 인수 조건 (Acceptance Criteria)

- [ ] 홈 화면에 "SDK Demos" 카테고리가 표시된다
- [ ] SDK Demos 카테고리를 탭하면 SDK 목록 화면으로 이동한다
- [ ] SDK 목록 화면에 "QnA", "Notice" 항목이 표시된다
- [ ] QnA 항목을 탭하면 QnA 질문 제출 화면(QnaSubmitView)이 표시된다
- [ ] Notice 항목을 탭하면 공지사항 목록 화면(NoticeListView)이 표시된다
- [ ] 각 SDK 화면에서 뒤로가기 버튼으로 이전 화면으로 복귀할 수 있다
- [ ] SDK 위젯들이 라이트/다크 테마를 올바르게 적용한다
- [ ] QnA 화면에서 입력 필드, 유효성 검증, 제출 버튼이 정상 동작한다
- [ ] Notice 화면에서 목록 표시, 카드 탭, 상세 화면 이동이 정상 동작한다
- [ ] SDK 패키지의 원본 코드는 수정하지 않는다 (데모 앱만 확장)
- [ ] 모든 화면이 GetX 라우팅 및 바인딩 패턴을 따른다

## 엣지 케이스

- **SDK 패키지 의존성 누락**: pubspec.yaml에 qna, notice 패키지가 없으면 SDK 목록 항목을 탭할 때 에러가 발생하지 않고, 안내 메시지를 표시한다
- **API 서버 미설정**: 서버 URL이 설정되지 않았거나 연결할 수 없는 경우에도 UI는 정상 표시되고, 에러 상태를 시각적으로 표시한다
- **빈 공지사항 목록**: 공지사항이 하나도 없을 때 빈 상태 화면이 표시된다
- **네트워크 타임아웃**: QnA 제출 또는 Notice 로드 중 타임아웃 발생 시 에러 모달/스낵바가 표시되고, 재시도 옵션을 제공한다
- **긴 공지 제목/내용**: 공지 제목이나 내용이 매우 긴 경우 카드와 상세 화면에서 텍스트 오버플로가 발생하지 않고 줄바꿈 또는 스크롤로 처리된다
- **테마 전환 중 SDK 화면 열림**: 테마 전환 애니메이션 중에도 SDK 화면이 올바르게 렌더링된다

## 비즈니스 규칙

- 데모 시스템은 개발 및 QA 환경에서만 사용한다 (프로덕션 앱에는 포함하지 않음)
- SDK 패키지의 원본 코드는 수정하지 않고, 데모 앱에서 패키지를 import하여 사용한다
- 각 SDK는 독립적으로 테스트 가능해야 한다 (QnA 테스트 시 Notice 의존성 불필요)
- SDK 데모는 실제 서버 연동과 모의 데이터 모두 지원 가능해야 한다 (환경 설정에 따라)
- 모든 SDK 데모 화면은 design_system_demo 앱의 기존 네비게이션 및 테마 시스템과 일관성을 유지한다
- SDK 데모 목록은 향후 새로운 SDK 추가 시 쉽게 확장 가능하도록 구조화한다

## 필요한 데이터

| 데이터 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| SDK 카테고리 목록 | List<SDKItem> | ✅ | 홈 화면에 표시할 SDK 항목 (name, icon, route) |
| QnA 화면 라우트 | String | ✅ | QnA 데모 화면 경로 |
| Notice 화면 라우트 | String | ✅ | Notice 데모 화면 경로 |
| 앱 테마 모드 | ThemeMode | ✅ | 라이트/다크 테마 상태 (SketchThemeController) |
| API 베이스 URL | String | ❌ | 서버 연동 시 필요 (환경 변수 또는 config) |
| 앱 코드 (appCode) | String | ❌ | SDK가 멀티테넌트 지원 시 필요 (예: "demo") |

## 비기능 요구사항

### 성능
- SDK 목록 화면 렌더링: 100ms 이내
- QnA/Notice 화면 초기 로드: 500ms 이내 (서버 응답 제외)
- 테마 전환 애니메이션: 300ms (부드러운 전환)

### 접근성
- 모든 버튼과 카드에 명확한 라벨 제공 (스크린 리더 지원)
- 터치 영역 최소 48x48 (Material Design 가이드라인)
- 충분한 색상 대비 (WCAG AA 레벨)

### 에러 처리
- 네트워크 에러: 사용자 친화적 메시지 + 재시도 옵션
- 패키지 미설치: 에러가 아닌 안내 메시지로 표시
- API 응답 에러: 에러 코드별 구체적인 메시지 제공 (400, 404, 500 등)
- 예외 발생 시: 앱이 크래시되지 않고 에러 화면 또는 스낵바로 피드백

### 유지보수성
- SDK 패키지 업데이트 시 데모 앱 코드 수정 최소화
- 새 SDK 추가 시 목록에 항목만 추가하면 동작하도록 구조화
- 코드 중복 최소화 (공통 레이아웃, 네비게이션 로직 재사용)
- 주석은 한글로 작성, 기술 용어는 영어 유지
